# Project Context

> This document provides high-level project direction to the bug pipeline.
> The Supervisor agent reads this file every cycle and uses it to keep all agents
> aligned with the project's goals, priorities, and constraints.

---

## Project Overview

**Name:** @oni.bot/core
**Type:** TypeScript library and SDK
**Stack:** TypeScript, Node.js 18+, Vitest, pnpm workspaces
**Description:** Graph, swarm, harness, tool, and platform primitives for production agent systems. The package owns reusable runtime infrastructure around agents: execution graphs, background sessions, guardrails, context/tool connectivity, orchestration, artifacts, and review gates.

---

## Current Priority

**Focus area:** Background-agent platform primitives and public API stability.
**Production hardening source:** See [PRODUCTION_HARDENING_PLAN.md](./PRODUCTION_HARDENING_PLAN.md) for the current readiness plan, release-gate backlog, and production completion sequence.
**Critical paths:**

```
- src/platform/       # task specs, sessions, triggers, identity/capability seams, artifacts, review gates
- src/harness/        # single-agent loop, external coding-agent adapters, session continuity
- src/swarm/          # multi-agent orchestration, queues, routing, scaling
- src/pregel/         # core graph execution engine
- src/guardrails/     # policy, budget, filters, audit
- src/events/         # lifecycle telemetry contracts
- packages/*/src/    # extension packages that must keep type/test compatibility
```

**Off-limits:**

```
- dist/               # generated by tsc
- node_modules/       # dependencies
- docs/superpowers/   # planning artifacts unless explicitly requested
- *.jsonl             # session/agent logs
```

## Quality Goals

- Public exports must compile through `pnpm run typecheck` and `pnpm run build`.
- Behavioral changes need focused Vitest coverage near the module being changed.
- `pnpm run coverage:quality` is a blocking production gate with global and module-specific floors for platform, MCP/LSP, external-agent, prebuilt, tool, GitHub, and auth-resolver surfaces.
- Run `pnpm test` for shared runtime changes; run package checks when package APIs or workspace config change.
- Keep runtime dependencies at zero for `@oni.bot/core`.
- Guardrails and platform policy should be enforced in code, not only prompt text.
- Generated artifacts, local credentials, and session logs must stay out of git.
- Release readiness should be judged against `PRODUCTION_HARDENING_PLAN.md` and its `verify:release` target, not only the default `verify` script.

## Architecture Notes

### Key Patterns

- `StateGraph` and Pregel execution are the macro runtime; harness is the single-agent loop; platform is the background-agent control plane.
- Provider-specific coding agents belong behind `ExternalAgentDriver` or platform `AgentSessionRunner` interfaces.
- Native harness loops can run as governed platform sessions through `createAgentLoopSessionRunner`; keep tool policy wrapping enabled unless a caller has an equivalent enforcement layer.
- Compiled ONI/Swarm skeletons can run as governed platform sessions through `createSwarmSessionRunner`.
- Provider CLI options that can bypass sandboxing or permissions must fail closed unless the caller explicitly opts into the `unsafe` driver policy.
- CLI-backed external-agent runs should keep bounded event/output retention, no-inherited-env platform execution, stdout/stderr secret redaction, and process-tree cleanup semantics; avoid reintroducing unbounded stdout/stderr accumulation or broad `process.env` inheritance for governed sessions.
- External-agent provider paths must honor session ownership. Validate `cwd`, Codex `addDirs`, and Claude Code `mcpConfig`/`worktree` before spawning provider processes.
- Platform trigger ingestion includes CLI, scheduled, GitHub webhook with HMAC signature verification, chat command, and dependency alert adapters.
- `@oni.bot/tools` must stay usable without importing root `dist`; platform integration in that package should use structural runtime policy hooks, not direct core imports that break package typecheck order.
- External-agent JSONL parsing should keep malformed frames observable and handle nested provider frame shapes; array-recursive extraction is intentional.
- Failed platform-run external agents should always produce failed-run diagnosis artifacts and only sanitized resume/provider metadata summaries.
- Platform operators should prefer `getHealthSnapshot()`, `getAuditSummary()`, `summarizePlatformHealth()`, and `summarizePlatformAudit()` over ad hoc parsing of raw session records.
- Platform lifecycle observability uses structural `logger` and OTel-compatible `tracer` config on `BackgroundAgentPlatform`; keep emitted data scoped to ids/counts/statuses rather than secrets or full provider payloads.
- GitHub review publication is available through `GitHubArtifactStore`; it publishes PR artifacts or issue/PR comments and can mirror enriched artifacts into JSON-file or in-memory stores.
- Replaceable provider seams are preferred over hard-coded cloud/vendor behavior.
- Keep session, artifact, audit, environment, identity, and capability data serializable.
- Platform audit events should expose capability summaries for observability without recording secret values.
- Public APIs are exported from their module index and from the root package when broadly useful.
- Keep `pnpm run smoke:exports`, `pnpm run typecheck:exports`, and `pnpm run pack:snapshot` green when adding or changing public exports or package file lists; they verify built subpaths, generated declarations, and publishable tarball contents.

### Known Debt

- `src/research/` is specified in `RESEARCH_SPEC.md` but not implemented.
- The platform module now ships in-memory providers plus JSON-file, SQLite, and Postgres durable stores, a GitHub artifact publisher, local and HTTP/Cerebro execution environment providers, runtime policy helpers, reusable tool policy wrappers, CLI/scheduled/GitHub-webhook/chat/dependency trigger adapters, an external-agent session runner, and a local `platform-smoke` CLI path.
- README/GUIDE coverage, `OPERATIONS_RUNBOOK.md`, and `PACKAGE_RELEASE_POLICY.md` should stay synchronized with new public subpaths, production platform behavior, and packaging policy.
- Lint currently exits cleanly but emits many warnings in tests.
- `verify:release` is green as of 2026-05-23 and includes blocking coverage quality thresholds, subpath runtime/type smoke, and publishable workspace package pack checks; keep it green before moving to platform backend and policy work.
- `docs/` is ignored by `.gitignore`, so tracked production docs should live at root or the ignore policy should be changed.

### External Dependencies

- Optional peer stores/checkpointers: SQLite, Redis, Postgres.
- Optional agent runtimes: Codex CLI, Claude Code CLI.
- Optional integration surfaces: MCP, LSP, A2A, ActivePieces-derived integrations.

## Agent Tuning Guidance

### Hunter

- Prioritize security, concurrency, and public API contract bugs in critical paths.
- Treat prompt-only policy enforcement as suspicious if a runtime enforcement point should exist.

### Fixer

- Keep fixes narrow and add regression tests.
- Preserve existing public API names unless a breaking change is explicitly requested.
- Do not rewrite generated output or dependency locks unless the change actually requires it.

### Validator

- Verify with the smallest focused test first, then broader commands when the change touches shared runtime or exports.
- Check package export/build behavior for new public modules.

### Security Auditor

- Prioritize path traversal, prompt injection in stored artifacts, command execution boundaries, secret exposure, SSRF, and prototype pollution.

### Supervisor

- Keep platform work aligned to the core abstraction: task plus trigger becomes a governed, isolated, observable session that produces a reviewable artifact.
